**버전:** v1.0
**작성일:** 2026-07-12
**작성자:** 정우 (with Claude)
**상태:** 구현 준비 완료 — Claude Code 단계별 프롬프트 작성 대기

## 1. 개요

### 1.1 한 줄 정의
친구들과의 모임에서 총무 한 명이 모든 결제를 담당한 뒤, **결제 내역을 입력하면 1인당 정산 금액을 계산하고, 피청구인별로 "감성 영수증" 이미지를 만들어 공유**할 수 있는 웹앱.

### 1.2 배경
- 현재 여우회 모임 정산은 스프레드시트로 수동 관리 중 (행 = 결제 항목, 열 = 멤버, 균등 분배, 인당 합계 + 총계).
- 스프레드시트 방식의 한계: 모바일에서 입력 불편, 공유 시 "얼마 보내면 돼?" 재질문 발생, 재미 요소 없음.
- 킥 포인트: **피청구인별 영수증** — 실제 감열지 영수증 스타일 + 하이라이트 사진 1장 첨부. 정산을 "청구"가 아니라 "추억 굿즈"로 바꾸는 것이 차별점.

### 1.3 타깃 사용자
| 역할 | 설명 | 주요 행동 |
|---|---|---|
| **총무 (청구인)** | 모임에서 모든 결제를 담당한 1인. 앱의 유일한 편집자 | 모임 생성, 결제 항목 입력, 영수증 생성·공유 |
| **피청구인** | 나머지 멤버 | 공유 링크로 정산 내역 열람(읽기 전용), 영수증 이미지 수신, 송금 |

### 1.4 배포 및 로드맵
- **1차:** 웹앱 — GitHub Pages, 단일 `index.html`, 빌드 도구 없음
- **2차 (본 PRD 범위 외):** 모바일 앱 출시 검토 → 이를 대비해 **모바일 퍼스트 반응형** + PWA 대응(manifest, 홈 화면 추가)을 1차부터 반영

## 2. 목표와 성공 기준

### 2.1 목표
1. 결제 항목 입력부터 전원 정산 금액 확정까지 **3분 이내** 완료
2. 피청구인이 "얼마를, 어디로, 왜 보내야 하는지"를 영수증 한 장으로 파악
3. 영수증이 카톡방에서 "귀엽다"는 반응을 얻어 자발적 공유가 일어나는 것

### 2.2 성공 기준 (v1)
- [ ] 여우회 실제 모임 1회에서 스프레드시트 없이 정산 완료
- [ ] 영수증 이미지가 카카오톡에서 깨짐 없이 표시
- [ ] 항목별 분배 금액 합계 = 결제 총액 (1원 오차 없음, 자동 검증 통과)
- [ ] 총무 기기 변경 후에도 텍스트 데이터 복원 (Firebase 동기화 확인)

## 3. 핵심 사용자 시나리오

### 시나리오 A: 당일 실시간 입력
1. 총무 정우가 모임 시작 전 "여우회 강화도 나들이" 모임 생성, 멤버 5인 등록
2. 결제할 때마다 항목 추가: 이름 + 이모지 + 금액 + 참여자 체크
3. 방탈출은 은찬이 안 해서 참여자에서 제외 → 4인 분배
4. 모임 종료 후 [정산하기] → 인당 금액 확인 → 멤버별 영수증 생성
5. 하이라이트 사진 1장 첨부 → 이미지 저장 → 카톡방에 5장 전송

### 시나리오 B: 사후 일괄 입력
1. 모임 다음 날 총무가 카드 내역 보며 일괄 입력
2. 더벤티는 정표가 "난 큰 사이즈 마셨으니 500원 더 낼게" → 금액 직접 조정
3. 나머지 흐름 동일

### 시나리오 C: 피청구인 열람
1. 은찬이 공유 링크 클릭 → 로그인 없이 정산 내역 열람 (읽기 전용)
2. 본인 열만 하이라이트된 표 + 본인 청구 금액 확인
3. 계좌번호 복사 버튼 → 송금

## 4. 기능 명세

### 4.1 모임(이벤트) 관리
- 모임 생성: 모임명, 날짜, 총무 계좌 정보(은행/계좌번호/예금주 — 선택 입력)
- 모임 목록: 최근순 정렬, 모임별 총액·정산 상태 뱃지 표시
- 모임 상태: `작성 중` → `정산 완료` (완료 후에도 수정 가능, 수정 시 상태 되돌림)
- 모임 복제: 멤버 구성을 그대로 가져와 새 모임 생성 (여우회처럼 고정 멤버 모임 대응)

### 4.2 멤버 관리
- 모임당 멤버 2~12인 (총무 포함)
- 멤버 속성: 이름, 대표 이모지/색상 (영수증·표에서 시각 구분용)
- 총무는 멤버 중 1인으로 지정 (기본값: 앱 사용자 본인)
- **총무 1인 고정** — 다중 결제자 상계 정산은 v2 백로그

### 4.3 결제 항목 입력
- 항목 속성:
  - 항목명 (필수) + 이모지 (선택, 이모지 피커 제공 — 스프레드시트의 📷🍦🚗☕ 습관 계승)
  - 결제 금액 (필수, 원 단위 정수)
  - 참여자 체크박스 (기본: 전원 선택)
  - 메모 (선택 — "예약금", "선불" 등)
- 분배 모드 (항목별로 선택):
  1. **균등 분배** (기본): 참여자 수로 N분할
  2. **금액 직접 조정**: 일부 참여자에게 고정 금액 지정 → 잔여 금액을 나머지 참여자가 균등 분배
- 항목 순서 변경(드래그), 수정, 삭제
- 입력 중 실시간으로 인당 누적 금액 미리보기

### 4.4 정산 계산 엔진

#### 분배 규칙
```
항목별:
  고정 금액 지정자 → 지정 금액 그대로
  나머지 참여자 → (결제 금액 − 고정 금액 합) ÷ 나머지 인원
```

#### 반올림 정책 (기본값, CONFIG로 변경 가능)
- 1인 분배액은 **1원 단위 내림(floor)**
- 내림으로 생기는 끝전 차액은 **총무가 부담** (총무 몫에 가산)
- 근거: 청구 금액이 "398.33원"처럼 나오는 것 방지 + 총무가 몇 원 더 내는 쪽이 관계상 자연스러움
- 대안 옵션(CONFIG): `10원 단위 올림`, `100원 단위 올림` (피청구인 몫 올림, 초과분은 총무 몫에서 차감)

#### 무결성 검증 (필수)
- **모든 항목에서 `Σ(참여자별 분배액) === 결제 금액`** 을 저장 시점마다 assert
- 불일치 시 저장 차단 + 어떤 항목이 문제인지 표시
- 고정 금액 합이 결제 금액을 초과하면 입력 단계에서 즉시 에러

#### 정산 결과 화면
- 스프레드시트와 동일한 구조의 표: 행 = 항목, 열 = 멤버, 우측 = 항목 합계, 하단 = 인당 총계 (파란 하이라이트 — 기존 습관 계승)
- 미참여 항목 셀은 `—` 표시
- 총무 몫은 "이미 결제함" 처리 → 피청구인별 **송금해야 할 금액**이 최종 산출물

### 4.5 영수증 생성 ⭐ 킥 포인트

#### 디자인 컨셉: 감열지 영수증 패러디
- 세로로 긴 영수증 비율 (카톡 이미지에 최적화, 권장 720×~1600px 가변)
- 구성 요소 (위→아래):
  1. 상호: `오늘은 내가 총무` + 지점명 패러디 (예: `여우회 강화도점` — 모임명 자동 삽입)
  2. 날짜/시간, 영수증 번호 (모임 ID 기반 장식용)
  3. **하이라이트 사진 1장** (선택) — 흑백/도트 톤 필터 옵션으로 감열지 감성 유지, 컬러 옵션도 제공
  4. 품목 리스트: 해당 피청구인이 **참여한 항목만** — `이모지 항목명 ..... 금액`
  5. 점선 구분선 + `합계` (본인 청구 금액, 큰 글씨)
  6. 총무 계좌: `은행 계좌번호 (예금주)` — 이미지에 포함
  7. 장식 바코드 + 하단 문구 (랜덤 멘트 풀: "오늘도 함께해줘서 고마워요", "교환·환불은 다음 모임에서" 등 — CONFIG 배열)
- 폰트: 도트/고정폭 계열 (예: Galmuri, DungGeunMo — 웹폰트 CDN)
- 절취선(지그재그 상하단 테두리) CSS 구현

#### 생성·공유 흐름
1. 정산 결과 화면에서 멤버 선택 → 영수증 미리보기
2. 사진 첨부: 파일 선택 → **클라이언트에서 리사이즈·압축** (긴 변 800px, JPEG 품질 0.7) → localStorage 저장
3. [이미지로 저장]: `html2canvas`(CDN)로 PNG 생성 → 다운로드
   - 모바일에서는 Web Share API(`navigator.share`) 지원 시 공유 시트 호출 → 카톡 직접 전송
4. [전체 영수증 저장]: 멤버 수만큼 순차 생성·다운로드
5. 계좌번호 복사 버튼 (`navigator.clipboard`)

#### 사진과 동기화의 관계 (중요한 설계 결정)
- 사진은 **총무 기기 localStorage에만 존재** → 영수증 이미지는 반드시 총무 기기에서 생성
- 피청구인이 공유 링크로 보는 웹 화면에는 사진이 없음 (텍스트 정산 내역만) — 사진이 담긴 완성본은 **이미지 파일로 전달**되는 것이 정상 흐름
- 총무가 기기를 바꾸면 사진은 유실되나 정산 데이터는 Firebase에서 복원됨 → UI에 "사진은 이 기기에만 저장됩니다" 안내 문구 1회 노출

### 4.6 데이터 저장 및 동기화 (하이브리드)

| 데이터 | 저장 위치 | 이유 |
|---|---|---|
| 모임, 멤버, 결제 항목, 정산 결과, 계좌 정보 | **Firebase Firestore** | 용량 작음, 기기 간 동기화·공유 링크에 필요 |
| 하이라이트 사진 (base64) | **localStorage** | 용량 큼 → Firebase Storage 비용·복잡도 회피 (개발 부담 최소화) |
| 로그인 세션 | Firebase Auth (Google) | mindpullness에서 검증된 패턴 재사용 |

- **쓰기 권한: 문서 생성자(총무)만** — Firestore Security Rules로 강제
- **읽기: 공유 링크 소지자 누구나** — 모임 문서에 `shareId`(무작위 문자열) 부여, `/#/view/{shareId}` 경로는 비로그인 읽기 전용
- 동기화 전략: 총무 단독 편집 구조이므로 **last-write-wins로 충분** (mindpullness의 다중 진입점 병합 문제와 달리 편집 주체가 1인) — 단, 저장은 debounce 800ms + 저장 상태 인디케이터("저장됨 ✓") 표시
- 오프라인 대응: Firestore 오프라인 퍼시스턴스 활성화 → 산속 펜션에서도 입력 가능, 온라인 복귀 시 자동 반영

### 4.7 화면 구성 (IA)

```
[홈] 모임 목록
 ├─ [모임 생성/설정] 모임명·날짜·계좌·멤버
 ├─ [결제 입력] 항목 리스트 + 추가 폼 + 인당 누적 미리보기
 ├─ [정산 결과] 스프레드시트형 표 + 인당 송금액
 │    └─ [영수증] 멤버별 미리보기 → 사진 첨부 → 이미지 저장/공유
 └─ [공유 뷰] /#/view/{shareId} — 읽기 전용 (비로그인)
```

- 라우팅: 해시 기반 (`location.hash`) — 단일 HTML 제약 준수
- 모바일 퍼스트: 하단 탭 or 상단 스텝 내비게이션, 표는 가로 스크롤 허용

## 5. 데이터 모델 (Firestore)

```
events (컬렉션)
  {eventId}
    ownerUid: string          // 총무 Auth UID
    shareId: string           // 공유용 무작위 ID (읽기 전용 링크)
    name: string              // "여우회 강화도 나들이"
    date: string              // "2026-07-12"
    status: "draft" | "settled"
    account: { bank, number, holder }   // 선택
    members: [ { id, name, emoji, color, isOwner } ]
    items: [
      {
        id, name, emoji, amount,        // amount: 정수(원)
        participantIds: [memberId],
        fixedAmounts: { memberId: 정수 } // 금액 직접 조정 시에만
        memo, order
      }
    ]
    createdAt, updatedAt
```

```
localStorage (총무 기기)
  onct_photos: { "{eventId}:{memberId}": base64JPEG }   // 영수증 사진
  onct_prefs:  { rounding: "floor1" | "ceil10" | "ceil100", ... }
```

- 정산 결과는 **저장하지 않고 항상 재계산** (단일 진실 원천 = items) → 데이터 불일치 원천 차단

## 6. 기술 스택 및 제약

- **단일 `index.html`** — HTML/CSS/Vanilla JS, 빌드 도구·npm 없음
- CDN: Firebase SDK(compat), html2canvas, 웹폰트(Galmuri 등)
- 모든 상수는 `CONFIG` 객체에 집약 (반올림 정책, 영수증 멘트 풀, 색상 팔레트, 이미지 압축 파라미터, Firebase 설정)
- 주석은 한국어
- 배포: GitHub Pages
- PWA: `manifest.json` + 아이콘 (서비스워커는 v1 범위 외 — Firestore 오프라인 퍼시스턴스로 갈음)

## 7. 비기능 요구사항

- **성능:** 항목 30개 × 멤버 12인에서 표 렌더링·재계산 즉시(<100ms)
- **접근성:** 금액은 `toLocaleString('ko-KR')` + ₩ 표기, 터치 타깃 44px 이상
- **개인정보:** 계좌번호는 총무가 자발 입력한 정보이며 공유 링크·영수증에 노출됨을 입력 시 안내. 이름은 실명 대신 별명 사용 권장 문구
- **브라우저:** 모바일 Safari/Chrome 최신 2개 버전 (카톡 인앱 브라우저 포함 — Web Share API 미지원 시 다운로드로 폴백)

## 8. 엣지 케이스 및 정책

| 케이스 | 정책 |
|---|---|
| 고정 금액 합 > 결제 금액 | 입력 단계 에러, 저장 차단 |
| 고정 금액 합 = 결제 금액 (잔여 0원) | 허용 — 나머지 참여자 분배액 0원 |
| 참여자 1인(총무만)인 항목 | 허용 — 청구 발생 없음, 표에는 기록 |
| 참여자 0인 | 저장 차단 |
| 금액 0원 항목 | 허용 (기록용, 예: 무료 입장) |
| 총무가 참여하지 않은 항목 | 허용 — 총무 분배액 0, 전액 피청구인 분배 |
| 예약금 + 잔금 (방탈출 사례) | v1: 별도 항목 2개로 입력 (스프레드시트 방식 유지). 항목 그룹핑은 v2 백로그 |
| localStorage 용량 초과 | 저장 실패 시 "사진 용량 초과 — 기존 모임 사진을 정리해주세요" 안내 + 사진 관리 화면 제공 |
| 공유 링크 유출 | shareId는 추측 불가 무작위 20자. 민감도 낮은 데이터이나 링크 재발급 기능 제공 |

## 9. 구현 단계 (Claude Code 단계별 프롬프트 기준)

| 단계 | 범위 | 완료 기준 |
|---|---|---|
| **Stage 1** | 뼈대: 해시 라우팅, CONFIG, 모임 CRUD + 멤버 관리 (localStorage만) | 모임 생성→멤버 등록→목록 표시 |
| **Stage 2** | 결제 항목 입력 + 정산 엔진 + 무결성 검증 + 결과 표 | 첨부 스프레드시트 데이터 재현 시 총계 ₩199,100 / ₩274,000 일치 |
| **Stage 3** | 영수증: 디자인, 사진 첨부·압축, html2canvas 저장, Web Share | 카톡 전송 화질 확인 |
| **Stage 4** | Firebase: Auth, Firestore 동기화, Security Rules, 공유 뷰 | 타 기기 복원 + 비로그인 열람 |
| **Stage 5** | 마감: PWA manifest, 반응형 다듬기, 엣지 케이스, 안내 문구 | 성공 기준(§2.2) 전체 통과 |

> Stage 2의 완료 기준에 **실제 스프레드시트 데이터를 회귀 테스트로 사용** — 두 이미지의 값을 시드 데이터로 하드코딩한 테스트 함수 포함 권장.
